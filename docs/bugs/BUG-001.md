# BUG-001 — Reloading an authenticated route bounces back to `/login`

**Status:** Fixed
**Severity:** High
**Opened:** 2026-05-17
**Source:** [button-link-audit-2026-05-17.md §High Priority Findings #1](../button-link-audit-2026-05-17.md)
**Area:** auth / session

## Summary

After a successful sign-in, refreshing any guarded route (`/weekend`,
`/itinerary`, `/activities`, `/restaurants`, `/saved`, `/events`, `/errand`,
`/profile`) sends the user back to `/login?returnUrl=<path>` even though the
persisted JWT is still valid. The route guard runs before
`SessionStore.rehydrate()` completes, so `isAuthenticated()` reads `false` at
guard-check time.

## Steps to reproduce

1. Open <https://gentle-rock-070070c0f.7.azurestaticapps.net/login>.
2. Sign in as `quinntynebrown@gmail.com` / seeded password — land on `/weekend`.
3. Press the browser refresh button while on `/weekend`.
4. Observe the network panel: `GET /api/auth/me` returns HTTP 200 with the user.
5. Observe the URL bar: the SPA navigates to `/login?returnUrl=%2Fweekend`.

## Expected

Reloading any authenticated route keeps the user on that route. The persisted
token in `localStorage['sd.auth.token']` and the HTTP 200 from `/api/auth/me`
together mean the session is still valid; the guard must wait for rehydration
before deciding.

## Actual

`requireAuth` reads `SessionStore.isAuthenticated()` synchronously, gets
`false` while `rehydrate()` is still inflight, and redirects to `/login`. The
subsequent `/api/auth/me` 200 is then wasted.

## Likely source

- `frontend/projects/saturdaze/src/app/auth/require-auth.guard.ts`
- `frontend/projects/saturdaze/src/app/app.ts`
- `frontend/projects/api/src/lib/services/session-store.ts`

## Acceptance

- An app initializer (`provideAppInitializer` / `APP_INITIALIZER`) awaits
  `SessionStore.rehydrate()` before router activation; OR the guard returns an
  `Observable<boolean>`/`Promise<boolean>` that resolves after rehydration.
- Playwright spec: sign in → land on `/weekend` → reload → still on `/weekend`,
  no `/login` redirect, no `returnUrl` query.
- The reload path issues at most one `/api/auth/me` call.

## Notes

The e2e suite worked around this race in `e2e/tests/sign-out.spec.ts` by
navigating to `/profile` via the bottom-nav inside the SPA. Once this bug is
fixed the workaround can be removed.
