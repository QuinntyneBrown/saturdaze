# BUG-006 — `POST /api/auth/forgot-password` returns 404

**Status:** Fixed
**Severity:** High
**Opened:** 2026-05-17
**Source:** [button-link-audit-2026-05-17.md §High Priority Findings #3](../button-link-audit-2026-05-17.md)
**Area:** auth / backend

## Summary

The "Send reset link" button on `/forgot-password` posts to
`https://sd-api-uofnt2.azurewebsites.net/api/auth/forgot-password` and
receives HTTP 404. The deployed `AuthController` only exposes
`POST /register`, `POST /login`, `GET /me`. The endpoint and its
application handler need to be implemented.

## Steps to reproduce

1. Open <https://gentle-rock-070070c0f.7.azurestaticapps.net/forgot-password>.
2. Enter any email address and click **Send reset link**.
3. Open DevTools → Network and observe:
   `POST /api/auth/forgot-password` → 404 Not Found.

## Expected

A request for a forgot-password token returns HTTP 200 (or 202 Accepted),
always — regardless of whether the email is on record — to avoid account
enumeration. A reset token is persisted and an email (or dev-mode capture)
is dispatched.

## Actual

The endpoint is not registered. The client falls through to its catch-all
"navigate to /check-email" anyway, masking the failure (see
[BUG-010](BUG-010.md)).

## Likely source

- `backend/src/Saturdaze.Api/Controllers/AuthController.cs`
- `backend/src/Saturdaze.Application/Auth/ForgotPassword/*` (to be added)
- `backend/src/Saturdaze.Infrastructure/...` (token persistence, email sender)

## Acceptance

- `POST /api/auth/forgot-password` accepts `{ "email": "..." }` and returns
  200/202 regardless of whether the email exists.
- A `PasswordResetToken` row is persisted with `userId`, `tokenHash`,
  `expiresUtc` (TTL ≤ 60 minutes), `consumedUtc IS NULL`.
- An email is sent (or dev-mode log/file capture) containing the reset link
  `<frontend>/reset-password?token=<token>`.
- Integration test in `Saturdaze.Api.Tests` covers happy path + unknown
  email (both 200) and rate-limit/abuse if implemented.
