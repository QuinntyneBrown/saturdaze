# BUG-010 — `/forgot-password` navigates to `/check-email` even on a 404 transport failure

**Status:** Fixed
**Severity:** High
**Opened:** 2026-05-17
**Source:** [button-link-audit-2026-05-17.md §High Priority Findings #3](../button-link-audit-2026-05-17.md)
**Area:** auth / frontend

## Summary

Clicking **Send reset link** on `/forgot-password` always navigates to
`/check-email` after the network call, even when the call fails with a
transport / configuration error (currently HTTP 404 because the endpoint
isn't implemented). The result is a fake success — the user believes a
reset email was sent when nothing happened.

Account-existence privacy means we should *not* reveal whether the email
matched a user, but a 404 / 5xx / network error is a *configuration*
failure, not a "this email doesn't exist" signal, and should not be hidden.

## Steps to reproduce

1. Open <https://gentle-rock-070070c0f.7.azurestaticapps.net/forgot-password>.
2. Enter any email and click **Send reset link**.
3. DevTools → Network: `POST /api/auth/forgot-password` → 404.
4. URL bar: app navigates to `/check-email`.
5. UI gives no hint that the request failed.

## Expected

- HTTP 2xx → navigate to `/check-email` with the destination email in
  router state (privacy-preserving: same response shape whether or not the
  email exists).
- HTTP 4xx (except known business-logic 4xx like rate-limit) / 5xx /
  network error → stay on `/forgot-password`, surface an inline error
  ("Couldn't send the reset link — try again in a minute"), and log the
  failure for monitoring.

## Actual

Page navigates to `/check-email` unconditionally; transport failures are
swallowed.

## Likely source

- `frontend/projects/saturdaze/src/app/pages/forgot-password/forgot-password.page.ts`
- `frontend/projects/api/src/lib/services/auth.service.ts`
  (`forgotPassword`)

## Acceptance

- Forgot-password handler navigates to `/check-email` only on a 2xx
  response.
- Non-2xx responses show an inline error and do not navigate.
- Playwright spec mocks a 500 from `/api/auth/forgot-password` and asserts
  the page stays put with an error visible.

## Notes

Once [BUG-006](BUG-006.md) lands, the 404 will become a 2xx and the visible
symptom disappears — but the underlying client logic must still distinguish
2xx from non-2xx, so don't close this bug just by fixing the backend.
