# BUG-038 — `/restaurants` "Lock it in" has no action

**Status:** Fixed
**Severity:** Medium
**Opened:** 2026-05-17
**Source:** [button-link-audit-2026-05-17.md §Restaurants](../button-link-audit-2026-05-17.md)
**Area:** restaurants

## Summary

The **Lock it in** button on each restaurant card produces no visible
state change. Selecting a restaurant for the weekend plan is a key
product action and is not wired.

## Steps to reproduce

1. Sign in and open <https://gentle-rock-070070c0f.7.azurestaticapps.net/restaurants>.
2. Click **Lock it in** on any card.
3. Observe: no chip change, no API call, no toast.
4. Reload — no persisted "locked" state visible.

## Expected

Click persists the selection into the active weekend plan:

- Calls `POST /api/weekends/current/restaurant` (or similar) with the
  restaurant id and the meal slot.
- Card visibly becomes the "locked" state (chip, disabled vote buttons,
  reduced opacity on siblings).
- The weekend itinerary picks up the choice on next view.

## Actual

Inert button.

## Acceptance

- Click persists the lock and reflects it visually.
- Selection survives reload.
- The selected restaurant appears in the weekend itinerary.
