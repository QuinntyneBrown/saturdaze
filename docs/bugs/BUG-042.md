# BUG-042 — `/saved` "Repeat" has no action

**Status:** Fixed
**Severity:** Medium
**Opened:** 2026-05-17
**Source:** [button-link-audit-2026-05-17.md §Saved](../button-link-audit-2026-05-17.md)
**Area:** saved

## Summary

The **Repeat** button on each saved-weekend card produces no visible
state change. Reusing a saved weekend as the current weekend plan is a
core saved-list product feature and is unimplemented.

## Steps to reproduce

1. Sign in and open <https://gentle-rock-070070c0f.7.azurestaticapps.net/saved>.
2. Click **Repeat** on any saved card.
3. Observe: no route, no dialog, no API call.

## Expected

Click instantiates the current weekend plan from the saved one — same
activities, same restaurants, same locks — and routes to `/weekend`.
A confirm dialog protects against overwriting an in-progress current
plan.

## Actual

Inert button.

## Likely source

- `frontend/projects/saturdaze/src/app/pages/saved/saved.page.html`
- `frontend/projects/saturdaze/src/app/pages/saved/saved.page.ts`
- `frontend/projects/api/src/lib/services/saved.service.ts`
- `frontend/projects/api/src/lib/services/weekend-plan.service.ts`

## Acceptance

- Click (after confirm) overwrites the current weekend with the saved one.
- User lands on `/weekend` and sees the saved plan as the current one.
- The saved weekend record is unchanged.
- E2E spec covers happy path + cancel.
