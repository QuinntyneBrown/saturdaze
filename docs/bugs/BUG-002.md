# BUG-002 — Login ignores `returnUrl` and always navigates to `/weekend`

**Status:** Fixed
**Severity:** High
**Opened:** 2026-05-17
**Source:** [button-link-audit-2026-05-17.md §High Priority Findings #1](../button-link-audit-2026-05-17.md)
**Area:** auth / session

## Summary

When an anonymous user tries to open a protected route, `requireAuth`
correctly redirects to `/login?returnUrl=<path>`. After signing in, however,
`LoginPage.submit()` ignores the `returnUrl` query parameter and routes the
user to `/weekend`. This breaks deep links, share links, and the
sign-out-and-back-in flow.

## Steps to reproduce

1. Sign out (or open in a clean private window).
2. Navigate directly to <https://gentle-rock-070070c0f.7.azurestaticapps.net/restaurants>.
3. Confirm the URL becomes `/login?returnUrl=%2Frestaurants`.
4. Sign in with seeded credentials.
5. Observe the URL bar after sign-in succeeds.

## Expected

After sign-in completes, the app routes to the value of `returnUrl` (here:
`/restaurants`). If `returnUrl` is missing or unsafe (off-origin), fall back
to `/weekend`.

## Actual

The app lands on `/weekend` regardless of `returnUrl`. The user has to
navigate manually back to `/restaurants`.

## Likely source

- `frontend/projects/saturdaze/src/app/pages/login/login.page.ts`
- `frontend/projects/saturdaze/src/app/auth/require-auth.guard.ts` (writes
  `returnUrl`, source of truth for the query param name)

## Acceptance

- `LoginPage.submit()` reads `ActivatedRoute.snapshot.queryParamMap.get('returnUrl')`
  and calls `router.navigateByUrl(returnUrl ?? '/weekend')`.
- Only same-origin absolute paths starting with `/` are honored — anything
  else falls back to `/weekend` (prevent open-redirect).
- Playwright spec: anonymous open of `/restaurants` → login bounce → sign in →
  lands on `/restaurants`.

## Notes

The `returnUrl` query param is the same one the `authInterceptor` writes on
401 (`auth/auth.interceptor.ts`). Keep the name consistent across all writers.
