# BUG-011 — Profile dialogs render without visible action buttons

**Status:** Fixed
**Severity:** High
**Opened:** 2026-05-17
**Source:** [button-link-audit-2026-05-17.md §High Priority Findings #4](../button-link-audit-2026-05-17.md)
**Area:** dialogs / components

## Summary

On the deployed Static Web App, every profile-level dialog opens with its
title and body content but **no visible action buttons**. The user is
trapped in a dialog they can only dismiss via Esc or backdrop click. The
root cause is that the `sd-dialog` shell projects `<ng-content>` slots from
inside an `@if`/`@else` control-flow block — Angular's compile-time control
flow drops the slot bindings, so children with `slot="actions"` never reach
the DOM.

Affected dialogs:

- `/profile` → **Add family member**
- `/profile` → edit family member
- `/profile` → **Add a commitment**
- `/profile` → edit commitment
- `/profile` → **Sign out**

## Steps to reproduce

1. Sign in and open <https://gentle-rock-070070c0f.7.azurestaticapps.net/profile>.
2. Click the **Add family member** "+" row.
3. Observe: the sheet opens with title + form fields but no `Cancel` /
   `Add member` buttons visible. Only a hidden submit button exists.
4. Press Esc to dismiss (no other way to close).
5. Repeat for **Sign out** in the Account section — same symptom.

## Expected

Every dialog renders its actions slot with the buttons the dialog component
emits. Each dialog can be confirmed and cancelled from the visible UI.

## Actual

Actions slot is empty. The user can only dismiss by Esc / backdrop click;
they cannot submit.

## Likely source

- `frontend/projects/components/src/lib/dialog/dialog.html`
- `frontend/projects/components/src/lib/dialog/dialog.scss`
- `frontend/projects/saturdaze/src/app/dialogs/family-member-dialog/family-member-dialog.html`
- `frontend/projects/saturdaze/src/app/dialogs/commitment-dialog/commitment-dialog.html`
- `frontend/projects/saturdaze/src/app/dialogs/sign-out-dialog/sign-out-dialog.html`

## Acceptance

- `sd-dialog` template renders `<ng-content select="[slot=actions]" />`
  unconditionally (outside any `@if`/`@else`); CSS `order` restores visual
  sequence if the projected slot precedes the default `<ng-content>`.
- All five dialogs above show visible primary + secondary action buttons.
- Playwright spec for each profile dialog asserts the action buttons are
  visible and can confirm / cancel from the rendered UI (not just by Esc).

## Notes

- The working tree contains uncommitted changes to `dialog.html` and
  `dialog.scss` that move the projection slots out of the control-flow
  block (CSS `order` 1/2/3/4 reorders grip → header → content → actions).
  Verify those changes deploy correctly before closing this bug.
- The `sd-dialog` `[static]` host attribute still needs to work for the
  `/dialogs` gallery review surface.
