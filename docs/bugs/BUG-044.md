# BUG-044 — `/errand` "Add to weekend" has no action

**Status:** Fixed
**Severity:** Medium
**Opened:** 2026-05-17
**Source:** [button-link-audit-2026-05-17.md §Errand](../button-link-audit-2026-05-17.md)
**Area:** errand

## Summary

The **Add to weekend** primary CTA on `/errand` produces no visible state
change, no API call, and no navigation. The form fields above it are
also not bound to state, so even if a backend endpoint existed there is
nothing to submit.

## Steps to reproduce

1. Sign in and open <https://gentle-rock-070070c0f.7.azurestaticapps.net/errand>.
2. Fill the errand form fields with arbitrary values.
3. Click **Add to weekend**.
4. Observe: nothing — no API call, no route change, no toast, no error.

## Expected

- The form fields are bound to a `signal` or `FormGroup` and the submit
  button is disabled until the form is valid.
- Click posts the errand to `POST /api/weekends/current/errands` (or
  equivalent), persists it as a weekend block, and routes the user back
  to `/weekend` or `/itinerary` with a confirmation toast.

## Actual

Form fields are unbound; submit button is inert.

## Likely source

- `frontend/projects/saturdaze/src/app/pages/errand/errand.page.html`
- `frontend/projects/saturdaze/src/app/pages/errand/errand.page.ts`
- Backend: errand persistence + weekend-block integration (likely
  missing).

## Acceptance

- Form fields are bound to state with validation.
- Submit disabled when invalid; enabled when valid.
- Submit POSTs and the resulting weekend block appears on `/weekend` /
  `/itinerary`.
- E2E spec covers the full add flow.
