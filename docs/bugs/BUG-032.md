# BUG-032 — `/itinerary` footer "Lock day" has no action

**Status:** Fixed
**Severity:** Medium
**Opened:** 2026-05-17
**Source:** [button-link-audit-2026-05-17.md §Itinerary](../button-link-audit-2026-05-17.md)
**Area:** itinerary

## Summary

The footer **Lock day** button on `/itinerary` produces no visible state
change. Locks at day granularity (preserve the entire day on the next
regenerate) are not implemented.

## Steps to reproduce

1. Sign in and open <https://gentle-rock-070070c0f.7.azurestaticapps.net/itinerary>.
2. Click **Lock day** in the footer.
3. Observe: no API call, no toast, no chip change.

## Expected

Click toggles the lock state for the currently active day:

- Calls `POST /api/weekends/current/days/{day}/lock` (or a `PATCH` toggle).
- Updates a visible "Locked" chip / icon on the day switcher.
- Subsequent regenerates preserve the locked day's blocks.

## Actual

Inert button. Lock state is invisible to the user.

## Likely source

- `frontend/projects/saturdaze/src/app/pages/itinerary/itinerary.page.html`
- `frontend/projects/saturdaze/src/app/pages/itinerary/itinerary.page.ts`
- `frontend/projects/api/src/lib/services/weekend-plan.service.ts`

## Acceptance

- Click toggles lock state via API.
- Locked day shows a visible affordance (chip / icon).
- A subsequent regenerate preserves the locked day's blocks.
- E2E spec covers "lock → regenerate → blocks preserved" loop.
