# BUG-009 — `/check-email` "Resend" is a no-op without a stored email

**Status:** Fixed
**Severity:** High
**Opened:** 2026-05-17
**Source:** [button-link-audit-2026-05-17.md §High Priority Findings #3 + §Check Email](../button-link-audit-2026-05-17.md)
**Area:** auth / frontend

## Summary

The "Resend" button on `/check-email` issues no network request and the page
flashes "Resent — check your inbox again" anyway. The page has no context
about *which* email to resend to — the destination email is not stored or
carried through the forgot-password / signup → check-email transition.

## Steps to reproduce

1. Open <https://gentle-rock-070070c0f.7.azurestaticapps.net/check-email>
   directly (no signed-in user, no prior forgot-password submission).
2. Click **Resend**.
3. DevTools → Network: no request is sent.
4. The UI shows "Resent — check your inbox again".

## Expected

One of:

- The page is reachable only via a redirect that carries the email (router
  state or query param), and **Resend** calls the matching backend endpoint
  (`forgot-password` or `resend-verification`) for that email.
- If the page is opened without context, prompt for the email first and
  **Resend** is disabled until one is entered.

Either way, the toast/banner only shows after a successful network response.

## Actual

Click handler short-circuits to a fake success regardless of state.

## Likely source

- `frontend/projects/saturdaze/src/app/pages/check-email/check-email.page.ts`
- `frontend/projects/saturdaze/src/app/pages/check-email/check-email.page.html`
- `frontend/projects/saturdaze/src/app/pages/forgot-password/forgot-password.page.ts`
  (origin of the navigation that should carry the email)
- `frontend/projects/saturdaze/src/app/pages/signup/signup.page.ts`
  (the other entry into `/check-email`)

## Acceptance

- `/check-email` knows which email it relates to (via router state, query
  param `?email=`, or a re-fetch from `SessionStore.user`).
- **Resend** calls `forgot-password` or `resend-verification` depending on
  the source flow; success state only renders after HTTP 200/202.
- If no email is known, the button is disabled and the page shows an
  inline email entry.
- Playwright spec covers each entry path.
