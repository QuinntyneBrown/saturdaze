# BUG-007 — `POST /api/auth/reset-password` returns 404

**Status:** Fixed
**Severity:** High
**Opened:** 2026-05-17
**Source:** [button-link-audit-2026-05-17.md §High Priority Findings #3](../button-link-audit-2026-05-17.md)
**Area:** auth / backend

## Summary

The "Update password" button on `/reset-password?token=<value>` posts to
`/api/auth/reset-password` and receives HTTP 404. The page then shows a
generic error. The endpoint, command handler, and token consumption logic
are missing.

## Steps to reproduce

1. Open
   <https://gentle-rock-070070c0f.7.azurestaticapps.net/reset-password?token=valid-mock-reset-token>.
2. Enter a new password and click **Update password**.
3. DevTools → Network: `POST /api/auth/reset-password` → 404.
4. The UI displays a generic error.

## Expected

`POST /api/auth/reset-password` with `{ token, newPassword }`:

- Returns 200 with `{ accessToken, ... }` on a valid, unconsumed,
  non-expired token; consumes the token; updates the user's password hash.
- Returns 400 with a typed error code on invalid/expired/already-consumed
  tokens.
- Returns 400 if the new password fails complexity rules.

## Actual

Endpoint not registered → 404 → page shows a generic error and the user
cannot complete password reset.

## Likely source

- `backend/src/Saturdaze.Api/Controllers/AuthController.cs`
- `backend/src/Saturdaze.Application/Auth/ResetPassword/*` (to be added)
- `backend/src/Saturdaze.Domain/Entities/PasswordResetToken.cs` (or similar)

## Acceptance

- Endpoint live at `POST /api/auth/reset-password`.
- Valid token: 200, token row marked consumed, password hash (bcrypt)
  updated, user can log in with the new password and not the old one.
- Invalid/expired/consumed token: 400 with stable error code.
- Integration tests cover all three branches.
