# BUG-008 — `/verify-email` flow has no backing endpoint and always fails

**Status:** Fixed
**Severity:** High
**Opened:** 2026-05-17
**Source:** [button-link-audit-2026-05-17.md §High Priority Findings #3](../button-link-audit-2026-05-17.md)
**Area:** auth / backend

## Summary

Loading `/verify-email?token=<value>` triggers the page's automatic
verification flow. The backend has no `POST /api/auth/verify-email` (nor
`POST /api/auth/resend-verification`), so the call 404s and the page shows
"Verification didn't work" with a generic error. Email-verification soft-gate
features that depend on `emailVerifiedUtc` cannot exit their pending state.

## Steps to reproduce

1. Open
   <https://gentle-rock-070070c0f.7.azurestaticapps.net/verify-email?token=valid-mock-verify-token>.
2. Wait for the auto-verify request.
3. DevTools → Network: `POST /api/auth/verify-email` → 404.
4. The page displays "Verification didn't work".

## Expected

`POST /api/auth/verify-email` with `{ token }`:

- 200 on valid, unconsumed, unexpired token; sets `User.EmailVerifiedUtc`
  to `now`; consumes the token; returns the updated `UserDto`.
- 400 on invalid/expired/consumed tokens with a stable error code.

A companion `POST /api/auth/resend-verification` issues a new token for the
signed-in user (or for an emailed address while anonymous) and dispatches
the verification email.

## Actual

Endpoint not registered → 404 → all signed-up users remain
unverified-forever from the UI's perspective.

## Likely source

- `backend/src/Saturdaze.Api/Controllers/AuthController.cs`
- `backend/src/Saturdaze.Application/Auth/VerifyEmail/*` (to be added)
- `backend/src/Saturdaze.Application/Auth/ResendVerification/*` (to be added)
- `frontend/projects/saturdaze/src/app/pages/verify-email/verify-email.page.ts`

## Acceptance

- Both endpoints live; integration tests cover happy + failure paths.
- After a successful verify, `GET /api/auth/me` returns
  `emailVerifiedUtc != null`.
- The `/verify-email` page shows a success state and routes the user on.
