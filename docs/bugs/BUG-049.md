# BUG-049 — Bottom-nav re-overlaps Safari URL bar when scrolling up (BUG-047 fix was incomplete)

**Status:** Fixed
**Severity:** High
**Opened:** 2026-05-17
**Source:** User report (iPhone, mobile Safari) — repeat of [BUG-047](BUG-047.md)
**Area:** nav

## Summary

After [BUG-047](BUG-047.md) shipped (May 2026), the bottom-nav still gets overlapped by iOS Safari's URL/tab bar when the user **scrolls up** on a signed-in route. The earlier fix added `env(safe-area-inset-bottom)` to clear the home indicator and `viewport-fit=cover` to the viewport meta — but it did not account for Safari's **dynamic** URL bar, which expands to 80–150 px on scroll-up and is taller than the safe-area inset.

The root cause is that `position: fixed` elements with `viewport-fit=cover` are pinned to the *layout* viewport (the full screen), not the *visual* viewport (what's actually visible behind the chrome). When Safari's URL bar is expanded, the visible viewport bottom sits well above the layout viewport bottom, so a nav anchored 12 px above the layout bottom sits behind the chrome.

## Steps to reproduce

1. Open <https://gentle-rock-070070c0f.7.azurestaticapps.net/login> on iPhone Safari.
2. Sign in. Land on `/weekend`.
3. Scroll a small amount down, then **swipe back up** to the top of the page. Safari's URL bar expands to its full height.
4. Observe the bottom-nav: the lower half of the pill sits behind the URL bar / back button row.

## Expected

The bottom-nav remains fully visible above Safari's chrome at every scroll position, every iPhone form factor, and in PWA standalone mode.

## Actual

At scroll-top with Safari's URL bar fully expanded, the bottom-nav is partially or fully obscured.

## Root cause

The previous fix's `bottom: calc(12px + env(safe-area-inset-bottom, 0px))` only accounted for:

- ✅ The home indicator (`env(safe-area-inset-bottom)`, ~34 px on Face-ID).
- ❌ **Not** the dynamic Safari URL/tab bar (no CSS env var for this).

iOS Safari's URL bar height varies between 0 px (fully collapsed) and ~80–150 px (fully expanded). `env(safe-area-inset-bottom)` is the home indicator inset — a different thing — and stays constant.

## Fix

Change the `bottom` calc on `sd-bottom-nav` `:host` to take the **larger** of the home-indicator inset and the dynamic chrome height — they are not additive, since Safari's chrome overlays the home-indicator zone when visible:

```scss
bottom: calc(
  12px                                                           /* visual floor */
  + max(env(safe-area-inset-bottom, 0px), 100lvh - 100svh)       /* whichever obstruction is present */
);
```

`100lvh - 100svh` is exactly the chrome height in CSS pixels (0 when chrome is collapsed; the chrome's height when expanded). `max(…)` picks whichever obstruction is currently larger.

> **First-pass mistake noted for posterity.** The original commit (f5aa327) used `+` instead of `max(…)`, stacking the home-indicator inset on top of the chrome height. On real iOS Safari at scroll-top that produced a ~46 px gap above the visible viewport bottom — visually "way too high." Corrected the same day.

Codified in **[ADR-005](../adr/ADR-005-bottom-nav-device-chrome-clearance.md)** so future agents/developers don't lose the rule.

## Likely source (now fixed)

- `frontend/projects/components/src/lib/bottom-nav/bottom-nav.scss` — the `:host` `bottom` calc.

## Acceptance

- ✅ At iPhone Safari scroll-top with URL bar expanded, the bottom-nav pill is fully above the chrome.
- ✅ At scroll-down with chrome collapsed, the nav sits at its baseline 12 px above the home indicator.
- ✅ On Face-ID iPhone in PWA standalone mode (no Safari chrome), the nav clears the home indicator.
- ✅ Regression test (`e2e/tests/regression/bottom-nav-clearance.spec.ts`) asserts the three components are present in the source SCSS. Catches the regression at test time.

## Notes

The earlier BUG-047 fix dates from May 2026. This BUG-049 supersedes it: the SCSS comment, the ADR, and the regression test together make the *complete* clearance rule explicit. A future Claude session or developer simplifying the `bottom` calc must first re-read ADR-005.
