# BUG-024 — `/weekend` quick-action "Lock what's already perfect" has no action

**Status:** Fixed
**Severity:** Medium
**Opened:** 2026-05-17
**Source:** [button-link-audit-2026-05-17.md §Home](../button-link-audit-2026-05-17.md)
**Area:** weekend / home

## Summary

The **Lock what's already perfect** quick-action card on `/weekend`
produces no visible change. Locking individual weekend blocks (so that
subsequent regenerations preserve them) is a core product feature and
isn't wired.

## Steps to reproduce

1. Sign in and open <https://gentle-rock-070070c0f.7.azurestaticapps.net/weekend>.
2. Click the **Lock what's already perfect** quick-action card.
3. Observe: nothing — no route, no dialog, no state change.

## Expected

Click enters a "lock selection" mode on the weekend view (each block gains
a lock toggle), or routes to `/itinerary` in a lock-edit mode.

## Actual

Inert card.

## Likely source

- `frontend/projects/saturdaze/src/app/pages/home/home.page.html`
- `frontend/projects/saturdaze/src/app/pages/home/home.page.ts`

## Acceptance

- Click enters a lock-edit mode (or routes to one).
- A subsequent regenerate preserves the locked blocks (`weekend-plan`
  service exposes a `lockBlock(blockId)` / equivalent).
- E2E spec exercises the full "lock → regenerate → block preserved" loop.

## Notes

The `Locked` chip already shown on commitments in `/profile` suggests the
lock concept exists at the domain layer — wire it through.
