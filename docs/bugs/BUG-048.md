# BUG-048 — "Remember me" doesn't pre-fill email or invite the password manager on return

**Status:** Fixed
**Severity:** Medium
**Opened:** 2026-05-17
**Source:** User report (iPhone, mobile Safari)
**Area:** auth / session

## Summary

The sign-in form's "Remember me" toggle persists the JWT in
`localStorage` so the *session* survives a browser close, which is the
correct meaning of that toggle. But there's no signal to the browser's
password manager (no `autocomplete` or `name` attributes on the email
and password inputs) and no application-side memory of the last-used
email — so when a returning user lands on `/login`, the form is empty,
the keychain doesn't offer to fill it, and "Remember me" feels broken.

`sd-text-input` does not forward the `autocomplete` or `name` HTML
attributes from its host to the inner `<input>`, so even adding them in
the page templates would currently be a no-op.

## Steps to reproduce

1. On iPhone Safari, open
   <https://gentle-rock-070070c0f.7.azurestaticapps.net/login>.
2. Sign in with **Remember me** checked.
3. Sign out (or just close and reopen Safari).
4. Return to `/login`.

Expected: the email is pre-filled and Safari Keychain (or 1Password,
Chrome autofill, etc.) offers to fill the password.

Actual: both inputs are empty; the keychain never engages because the
inputs lack `autocomplete="username"` / `autocomplete="current-password"`.

## Expected

- When the user previously signed in with **Remember me** on, returning
  to `/login` pre-fills the email field with the last-used address and
  defaults the toggle to on.
- The browser password manager (Safari Keychain on iOS, Chrome/Edge
  autofill, third-party managers) recognises the form, offers to save
  on submit, and offers to fill on return.
- Unchecking **Remember me** before submitting forgets the email too —
  the toggle is the single user-visible signal for "trust this device".
- Signing out leaves the remembered email in place (sign-out forgets
  the session, not the device).
- Passwords are **never** stored in app-controlled storage — that stays
  the password manager's job for security reasons.

## Actual

- `<sd-text-input>` ignores `autocomplete` / `name` attributes; the
  generated `<input>` has neither.
- The login page never reads or writes a "last-used email" value, so
  there's nothing to pre-fill even after a successful sign-in.

## Likely source

- `frontend/projects/components/src/lib/text-input/text-input.ts`
  (no `autocomplete` / `name` inputs)
- `frontend/projects/components/src/lib/text-input/text-input.html`
  (no `[attr.autocomplete]` / `[attr.name]` bindings)
- `frontend/projects/saturdaze/src/app/pages/login/login.page.html`
  (no autocomplete hints on the email/password fields)
- `frontend/projects/saturdaze/src/app/pages/login/login.page.ts`
  (no pre-fill of email on mount)
- `frontend/projects/api/src/lib/services/session-store.ts`
  (no persistence of the remembered email next to the JWT)

## Acceptance

- The email field on `/login` is pre-filled when the user signed in
  with **Remember me** on within the last session, and the toggle
  defaults to on in that case.
- Submitting the form with **Remember me** off clears any previously
  remembered email.
- Submitting with **Remember me** on writes the entered email to
  `localStorage` under `sd.auth.remember.email`.
- `<input>` elements rendered by `<sd-text-input>` carry
  `autocomplete="username"` (email) and `autocomplete="current-password"`
  (password) on the sign-in page, plus matching `name` attributes, so
  iOS Safari Keychain and other password managers reliably engage.
- Existing visual snapshots for the login page continue to match —
  the change is attribute-only.

## Notes

This change deliberately does **not** persist passwords. Storing
credentials inside the SPA is a meaningfully different security
posture than letting the OS password manager handle them; we keep the
secure path and just make sure the browser can see the form clearly.

Signup and the password-reset flows could benefit from the same
`autocomplete` plumbing (`autocomplete="new-password"`, `email`,
`new-password` again on the reset screen), but that scope is out of
this bug and tracked separately if needed.
