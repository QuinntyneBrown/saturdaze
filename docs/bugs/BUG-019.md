# BUG-019 — `/check-email` "Open mail app" button has no action

**Status:** Fixed
**Severity:** Medium
**Opened:** 2026-05-17
**Source:** [button-link-audit-2026-05-17.md §Check Email](../button-link-audit-2026-05-17.md)
**Area:** check-email

## Summary

The **Open mail app** button on `/check-email` has no route, no
protocol-handler href (`mailto:`, `googlegmail://`, etc.), no dialog, and
no visible state change. It implies the page will deep-link the user into
their inbox, which is exactly what the user wants at that moment.

## Steps to reproduce

1. Open <https://gentle-rock-070070c0f.7.azurestaticapps.net/check-email>.
2. Click **Open mail app**.
3. Observe: nothing happens — URL unchanged, no protocol prompt, no native
   app launch.

## Expected

The button opens a mail surface — best-effort across providers:

- On mobile / OS-native: `href="mailto:"` triggers the default mail app.
- On desktop: a small menu / dropdown with Gmail / Outlook / Apple Mail
  web inbox links, or just a `mailto:` fallback.
- Or — if the deep-link UX isn't feasible — remove the button.

## Actual

Inert button.

- `frontend/projects/saturdaze/src/app/pages/check-email/check-email.page.html`

## Acceptance

- Click either opens a mail UX or the button is removed.
- Playwright spec asserts the chosen behaviour (the simplest implementation
  is an `<a href="mailto:">` — testable by reading the rendered `href`).
