# BUG-039 — `/saved` top-left "Back" has no action

**Status:** Fixed
**Severity:** Medium
**Opened:** 2026-05-17
**Source:** [button-link-audit-2026-05-17.md §Saved](../button-link-audit-2026-05-17.md)
**Area:** saved / nav

## Summary

The top-left **Back** icon on `/saved` produces no visible state change.
The user has no top-bar way to navigate out — they must rely on the
bottom-nav.

## Steps to reproduce

1. Sign in and open <https://gentle-rock-070070c0f.7.azurestaticapps.net/saved>.
2. Click the top-left back icon.
3. Observe: nothing — no `history.back()`, no route change.

## Expected

One of:

- The top bar uses `<sd-top-bar back>` (the same affordance used on
  `/profile`), which calls `Location.back()` and falls back to `/weekend`
  if there is no prior history entry.
- Or the leading icon button is explicitly wired to `Location.back()` with
  the same fallback.

## Actual

Inert icon button.

## Likely source

- `frontend/projects/saturdaze/src/app/pages/saved/saved.page.html`
- `frontend/projects/components/src/lib/top-bar/top-bar.ts`
  (verify the `back` attribute is honoured)

## Acceptance

- Top-left **Back** navigates back (or to `/weekend` as a fallback).
- E2E spec exercises both the "with history" and "no history" paths.
