# BUG-036 — `/restaurants` vote `Yes` / `No` buttons have no action

**Status:** Fixed
**Severity:** Medium
**Opened:** 2026-05-17
**Source:** [button-link-audit-2026-05-17.md §Restaurants](../button-link-audit-2026-05-17.md)
**Area:** restaurants / voting

## Summary

The **Yes** and **No** vote buttons on each restaurant card produce no
visible state change, do not toggle an active state, and do not call any
endpoint. The voting / consensus loop is unimplemented.

## Steps to reproduce

1. Sign in and open <https://gentle-rock-070070c0f.7.azurestaticapps.net/restaurants>.
2. Click **Yes** (or **No**) on any restaurant card.
3. Observe: button does not visually toggle. No API call.
4. Reload the page. The vote (if persisted) does not reappear.

## Expected

- Click sets the user's vote for that restaurant and visually toggles the
  active state on the chosen button (mutually exclusive with its sibling).
- A `POST /api/restaurants/{id}/vote` (or equivalent) persists the vote.
- Subsequent loads show the persisted vote.
- A consensus indicator (e.g. "Sara is in / out") reflects the family's
  votes when available.

## Actual

Inert buttons; no visual or persistent state.

- `frontend/projects/saturdaze/src/app/pages/restaurants/restaurants.page.html`

## Acceptance

- Clicking emits the vote, toggles UI state, persists, and survives reload.
- Vitest spec for `vote-row` covers toggle semantics.
- E2E spec covers persistence across reload.
