# L1 - High-Level Requirements

> Each L1 names a capability the Saturdaze system must deliver. Detailed, testable behaviors live in `L2.md`.

## L1-001: Authentication and Account Lifecycle
The system must let a family register, sign in, sign out, recover a forgotten password, reset it via emailed link, and verify ownership of their email address. Sessions must persist across page reloads when the user opts in, and must be revocable.

## L1-002: Family Profile and Preferences
Each authenticated account must own a single Family profile capturing home location, members (with ages), recurring commitments (weekly events with day/time), daily rhythm anchors, likes/dislikes, and toggleable preferences (budget, novelty, notifications). The profile must drive all downstream personalization.

## L1-003: Weekend Itinerary Generation
The system must generate a Saturday + Sunday itinerary for the family's upcoming weekend, composed of time-blocked activities, meals, drives, downtime, errands, and the family's recurring commitments. A "current weekend" must always be addressable, and any prior weekend must be retrievable by ID.

## L1-004: Itinerary Block Control
Users must be able to lock individual blocks (so regeneration leaves them alone), swap a block for a different suggestion (optionally rejecting prior suggestions), and regenerate the unlocked portion of the weekend. Locked blocks (including commitments) must never be displaced by automated regeneration.

## L1-005: Activity Discovery and Filtering
The system must present curated activities filtered by indoor/outdoor, age fit, drive time, and weather suitability, grouped to surface weather-appropriate picks first, rainy-day fallbacks second, and novelty options ("Try something new") third.

## L1-006: Restaurant Discovery and Picks
The system must surface restaurant picks for a given day and meal slot, filterable by wife-approved status, distance, and patio availability, with a featured top pick that supports voting by each family member and one-tap "lock it in" to the itinerary.

## L1-007: Local Events Discovery
The system must list local community events within driving distance of home, grouped by Saturday / Sunday / Coming-soon, and filterable by indoor/outdoor and category (Seasonal, Theatre, Festivals, etc.).

## L1-008: Errand Slot Suggestion
Users must be able to capture a shopping errand (description + estimated minutes) and have the system propose the best slot to fit it into the weekend, with the option to accept the suggestion, pick a different slot, or mark the errand done.

## L1-009: Weather Forecast for the Weekend
The system must display a daily forecast (high, low, condition, note) for Saturday and Sunday of the queried weekend, anchored to the family's home location. Forecast data must come from a reputable provider, must be cached to limit upstream load, and must degrade gracefully when the provider is unavailable.

## L1-010: Saved Weekend History
The system must persist every completed weekend, let users favourite weekends, rate them 1–5 stars, edit their title, and surface them in a history view with filters (All, Favourites, This year, 5★ only). Low-rated activities must be flagged as "avoid repeating" so future generations can skip them.

## L1-011: Responsive Web Experience
Every user-facing page must render correctly on extra-small (<576 px), small (≥576 px), medium (≥768 px), large (≥992 px), and extra-large (≥1200 px) viewports. The split-view pattern (sticky master / scrollable detail) must engage at the desktop breakpoint and collapse to single-column on mobile.

## L1-012: Security
The system must protect user credentials and session tokens, enforce authorization on all non-public endpoints, validate every external input, defend against the OWASP Top 10 (injection, broken auth, CSRF, etc.), and never log secrets or personally identifying information in plaintext.

## L1-013: Performance
API endpoints must respond within budgets appropriate to interactive use, frontend bundles must respect size budgets, weather and other slow upstream calls must be cached, and Angular route bundles must be lazy-loaded so first paint is not blocked by lower-traffic pages.

## L1-014: Accessibility
Every interactive element must be reachable and operable by keyboard, every meaningful element must expose an accessible name, color contrast must meet WCAG 2.1 AA, and dialogs must trap focus and announce themselves to assistive technology.

## L1-015: Observability and Resilience
The system must emit structured logs for every request, warn at startup when production-required configuration is missing, log upstream provider failures without crashing, and surface meaningful, non-leaky error messages to clients.

## L1-016: Deployment and Migration Pipeline
The system must deploy automatically to Azure on every push to `main`: build and publish the API to Azure App Service, apply EF Core migrations idempotently to Azure SQL, and publish the Angular frontend to Azure Static Web Apps with SPA navigation fallback.

## L1-017: Administrative CLI
A single CLI tool (`saturdaze`) must be the only supported entry point for administrative operations: applying migrations, seeding curated data (activities, restaurants, events, the Brown family, users), and resetting a development database. The CLI must accept a connection string by flag, environment variable, or configuration.

## L1-018: User-Contributed Local Events
Any authenticated user must be able to submit a local community event that isn't already in the system (e.g. a Port Credit street festival, a school fair, a one-off concert). Submissions enter a pending state and are not visible to other users until an administrator approves them. Submissions must support two entry points — a contextual floating "+" affordance on the events feed, and a dedicated "Submit an event" screen reachable from the events page. The submission form must capture title, date, and time as required, default the date and time to one hour from now rounded up to the next hour, and treat location, description, cost, age range, and source URL as optional. Approved events appear in the regular [L1-007](#l1-007-local-events-discovery) feed indistinguishable from curated entries; rejected events are removed and the submitter is notified.
