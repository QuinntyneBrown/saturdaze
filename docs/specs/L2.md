# L2 - Detailed Requirements

> Each L2 traces to one L1 in `L1.md`. Acceptance criteria use Given/When/Then so they translate directly to tests. Every acceptance test in this repo must carry a `Traces to: L2-NNN` header comment.

---

## L2-001: User signup creates account and family
**Traces to:** L1-001

The system must accept a signup payload (email, password, optional family name, optional home location), create a `User` row, create an empty `Family` row owned by that user, issue an access token + refresh token, and return both with the user record.

**Acceptance Criteria:**
1. Given no user exists with email `new@example.com`, when a `POST /api/auth/register` is sent with `{email: "new@example.com", password: "Passw0rd!"}`, then the response status must be 201 and the body must contain a non-empty `tokens.accessToken`, a non-empty `tokens.refreshToken`, and a `user` with `emailVerifiedUtc=null`.
2. Given a user already exists with normalized email `new@example.com`, when registration is retried with the same email, then the response must be 409 with error code `email_in_use`.
3. Given any successful registration, when the database is inspected, then exactly one new `Family` row must exist with the new user's `FamilyId` pointing to it.
4. Given a password shorter than 8 characters, when registration is attempted, then the response must be 400 with a validation error naming the password field.
5. Given the signup form on `/signup` with terms unchecked, when the user fills email/password, then the submit button must remain disabled.

## L2-002: User login returns tokens and rotates refresh
**Traces to:** L1-001

The system must authenticate a user by `Email` + `Password`, issue a fresh access token + refresh token on every successful login, and return a generic 400 error on any credential failure (without distinguishing "no such user" from "wrong password").

**Acceptance Criteria:**
1. Given a seeded user `quinntynebrown@gmail.com` with password `password123`, when `POST /api/auth/login` is sent with those credentials, then the response must be 200 with `tokens.accessToken`, `tokens.refreshToken`, and `user.email` matching the seeded user.
2. Given the same user, when login is sent with a wrong password, then the response must be 400 with error code `invalid_credentials` and the response body must not reveal whether the email exists.
3. Given a successful login, when the database is inspected, then a new `RefreshToken` row must exist with `ExpiresAtUtc` 14 days in the future and `RevokedAtUtc` null.

---

## L2-003: Sign-out revokes the active refresh token
**Traces to:** L1-001

A signed-in user must be able to sign out via a dialog from the profile page, after which their refresh token is revoked and stored token material is cleared from the client.

**Acceptance Criteria:**
1. Given the user is signed in, when they tap "Sign out" on `/profile` and confirm the dialog, then the client must clear access and refresh tokens from both `localStorage` and `sessionStorage`, and the next protected request must return 401.
2. Given an active refresh token at sign-out time, when the dialog is confirmed, then the corresponding `RefreshToken.RevokedAtUtc` must be set in the database.
3. Given the sign-out dialog is open, when the user presses Escape or taps the backdrop, then the dialog must close without signing the user out.

---

## L2-004: Forgot-password flow accepts an email
**Traces to:** L1-001

A signed-out user must be able to request a password-reset link by entering their email. The system must not reveal whether the email is registered.

**Acceptance Criteria:**
1. Given the user is on `/forgot-password`, when they submit a syntactically valid email, then they must be navigated to `/check-email` regardless of whether the email is registered.
2. Given an unregistered email is submitted, when the response is inspected, then it must be indistinguishable from a registered email submission (same status, same shape, same latency budget).
3. Given the email field is empty or malformed, when the user submits, then a field-level validation error must appear inline and the form must not be submitted.

---

## L2-005: Reset-password applies a new password via token
**Traces to:** L1-001

The system must accept a reset token plus a new password and confirmation, validate the token, enforce password complexity, and rotate any existing refresh tokens for that user on success.

**Acceptance Criteria:**
1. Given a valid, unexpired reset token, when the user submits matching passwords meeting complexity rules, then they must be redirected to `/login?reset=1` and the user's password hash must be updated.
2. Given an expired or invalid token, when the user submits, then an inline error must appear with a link back to `/forgot-password`, and the password hash must not change.
3. Given the password and confirmation fields differ, when the user submits, then a field-level "passwords do not match" error must appear and submission must be blocked.
4. Given a password shorter than 8 characters, or missing a number/symbol, when the user types, then an inline rule-checklist must indicate which rules are not yet satisfied.

## L2-006: Email verification links activate the account
**Traces to:** L1-001

The system must accept a verification token, mark the corresponding user as verified, and surface either a success or error state on `/verify-email`.

**Acceptance Criteria:**
1. Given a valid, unexpired verification token, when the user visits `/verify-email?token=<token>`, then `User.EmailVerifiedUtc` must be set to the current UTC time and the page must show a success state with primary CTA "Set up your family".
2. Given an invalid or expired token, when the user visits the page, then an error state must appear with a link to sign in or request a new link.
3. Given the user reaches the success state, when they tap "Skip — take me to the app", then they must be redirected to `/weekend`.

## L2-007: Remember-me toggle controls token persistence
**Traces to:** L1-001

The login form's "remember me" toggle must select between durable persistence (`localStorage`) and session-only persistence (`sessionStorage`) for the access and refresh tokens.

**Acceptance Criteria:**
1. Given remember-me is checked, when the user logs in successfully and then reloads the page, then the access token must still be present in `localStorage` and the user must remain on a protected route.
2. Given remember-me is unchecked, when the user logs in successfully, then `localStorage` must contain no Saturdaze token, `sessionStorage` must contain the access token, and closing the tab must end the session.

---

## L2-008: JWT-protected endpoints require a valid bearer token
**Traces to:** L1-001, L1-012

Every endpoint except `POST /api/auth/register`, `POST /api/auth/login`, `POST /api/auth/forgot-password`, `POST /api/auth/reset-password`, `POST /api/auth/verify-email`, `GET /api/weather` (no PII), and `GET /api/activities|/api/restaurants|/api/events` (read-only public data) must reject requests that lack a valid `Authorization: Bearer <jwt>` header.

**Acceptance Criteria:**
1. Given no Authorization header, when `GET /api/auth/me` is requested, then the response must be 401.
2. Given an expired access token, when any protected endpoint is requested, then the response must be 401.
3. Given a token signed with the wrong key, when any protected endpoint is requested, then the response must be 401.
4. Given a valid bearer token, when `GET /api/auth/me` is requested, then the response must be 200 with the caller's `UserDto`.

## L2-009: Family profile view returns the caller's family
**Traces to:** L1-002

`GET /api/family` must return the authenticated caller's family with all members, commitments, and preferences in one response.

**Acceptance Criteria:**
1. Given an authenticated user whose family has 4 members and 3 commitments, when `GET /api/family` is requested, then the response must include exactly those 4 members and 3 commitments.
2. Given an authenticated user, when `GET /api/family` is requested, then the response must include `homeLocation`, `budgetEnabled`, members (id, name, age), commitments (id, title, dayOfWeek, startTime, endTime), and preferences (id, kind, value).
3. Given no authenticated user, when `GET /api/family` is requested, then the response must be 401.

## L2-010: Family profile edit persists changes
**Traces to:** L1-002

`PUT /api/family` must accept a full or partial family profile and persist additions, edits, and removals to members, commitments, and preferences.

**Acceptance Criteria:**
1. Given an existing family with members `[Quinn, Sara]`, when `PUT /api/family` is sent with members `[Quinn, Sara, Eli]`, then a subsequent `GET /api/family` must return 3 members including Eli with the supplied age.
2. Given an existing commitment "Swim 9–10am Sat", when `PUT /api/family` is sent omitting that commitment, then it must be removed from the family.
3. Given the profile page on `/profile`, when the user taps "Add member" then a CDK dialog must open (no inline form), and saving must call `PUT /api/family` exactly once with the new member appended.
4. Given an invalid payload (e.g., member with negative age), when `PUT /api/family` is sent, then the response must be 400 with a validation error.

## L2-011: Recurring commitments produce locked blocks
**Traces to:** L1-002, L1-004

Every `Commitment` on the family profile must materialize as a locked itinerary block on every generated weekend at the commitment's day and time window.

**Acceptance Criteria:**
1. Given a family with commitment "Swim 9:00–10:00 Saturday", when a new weekend is generated, then the Saturday timeline must contain a block titled "Swim" from 09:00 to 10:00 with `IsLocked=true` and `Kind=Commitment`.
2. Given a regenerate is requested for an existing weekend, when the response is returned, then every commitment block from the prior weekend must remain in place at the same time.

## L2-012: Generate weekend creates a Saturday+Sunday plan
**Traces to:** L1-003

`POST /api/weekends/plan` must create a new `Weekend` row for the requested weekend date with blocks across Saturday and Sunday, including activities, meals, drives, downtime, and any commitments from the family profile.

**Acceptance Criteria:**
1. Given an authenticated user with a family profile, when `POST /api/weekends/plan` is sent with `weekendOf=<saturday-date>`, then the response must be 200 with a `WeekendDto` whose `weekendOf` matches the input and whose `blocks` array spans both Saturday and Sunday.
2. Given the same family generates a plan twice for the same weekend, when the database is inspected, then exactly one `Weekend` row must exist for that family + date pair (uniqueness enforced).
3. Given no family profile data, when generation is attempted, then the response must succeed but produce only the bare-minimum blocks the system can construct, and not throw a 500.

## L2-013: Current weekend endpoint returns the upcoming plan
**Traces to:** L1-003

`GET /api/weekends/current` must return the family's plan for the current or next upcoming Saturday, generating one on demand if absent.

**Acceptance Criteria:**
1. Given an authenticated user with no weekends, when `GET /api/weekends/current` is requested, then a new `Weekend` for the upcoming Saturday must be created and returned with status 200.
2. Given a weekend already exists for the upcoming Saturday, when `GET /api/weekends/current` is requested, then the response must reference the existing weekend's ID (not create a new row).

## L2-014: Block lock toggle is idempotent
**Traces to:** L1-004

`PUT /api/blocks/{id}/lock` must set `ItineraryBlock.IsLocked` to the request value and return the parent weekend.

**Acceptance Criteria:**
1. Given an unlocked block, when `PUT /api/blocks/{id}/lock` is sent with `{locked: true}`, then the response must be 200 with the weekend showing that block's `isLocked=true`.
2. Given an already-locked block, when the same request is sent again, then the response must still be 200 and the block must remain locked (no error).
3. Given a block ID that does not exist, when the endpoint is called, then the response must be 404.
4. Given a block that belongs to another family, when the caller is not a member of that family, then the response must be 403 or 404 (no information leak).

---

## L2-015: Block swap replaces a slot and respects rejected suggestions
**Traces to:** L1-004

`POST /api/blocks/{id}/swap` must replace the activity referenced by a block with the next-best candidate, optionally excluding a list of rejected activity IDs supplied by the client.

**Acceptance Criteria:**
1. Given an Activity block referencing Activity A, when `POST /api/blocks/{id}/swap` is sent with `rejectedActivityIds=[A]`, then the response must contain that same block ID but with a different `refId` (not A).
2. Given a locked block, when swap is attempted, then the response must be 409 with error code `block_locked` and the block must remain unchanged.
3. Given the only remaining candidate has already been rejected, when swap is attempted, then the response must be 200 and the block must keep its current activity (no-op) with a `reason` annotation.

---

## L2-016: Regenerate preserves locks and increments RegenerateCount
**Traces to:** L1-004

`POST /api/weekends/{id}/regenerate` must rebuild every unlocked block, leave every locked block in place, and increment `Weekend.RegenerateCount`.

**Acceptance Criteria:**
1. Given a weekend with 10 blocks of which 3 are locked, when regenerate is called, then exactly those 3 locked blocks (by ID and content) must appear unchanged in the response, and the other 7 must have new content.
2. Given a weekend with `RegenerateCount=2`, when regenerate is called, then the database must show `RegenerateCount=3`.

## L2-017: Activity list applies filters and excludes age-misfits
**Traces to:** L1-005

`GET /api/activities` must accept query filters (`indoor`, `maxDriveMinutes`, `minAge`, `maxAge`, `weather`, `tryNew`, `take`) and return activities whose attributes satisfy every supplied filter.

**Acceptance Criteria:**
1. Given seeded activities, when `GET /api/activities?indoor=true` is requested, then every item in the response must have `indoor=true`.
2. Given seeded activities including one with `maxAge=8`, when `?minAge=9` is requested, then that activity must not appear in the response.
3. Given `?take=3`, when the request is sent, then the response must contain at most 3 items.
4. Given the activities page on `/activities`, when the user taps the "Outdoor" filter chip, then only outdoor activities must be displayed within the visible viewport without a full page reload.

## L2-018: Activities page groups results into three sections
**Traces to:** L1-005

The `/activities` page must render three labelled sections in order: weather-fit picks for the current forecast, "If weather turns" fallbacks, and "Try something new" novelty picks.

**Acceptance Criteria:**
1. Given the activities page is loaded, when the DOM is inspected, then exactly three `sd-section` elements must appear in document order with titles matching the three section names.
2. Given the forecast for Saturday is sunny, when the page loads, then the weather-fit section must contain at least one outdoor activity and at most one indoor activity (outdoor-first ordering).

## L2-019: Restaurants list scopes by day and meal slot
**Traces to:** L1-006

`GET /api/restaurants` must require a `day` and `slot` parameter, optionally narrow by `nearActivityId`, and default `wifeApprovedOnly=true`.

**Acceptance Criteria:**
1. Given a request without `day` or `slot`, when `GET /api/restaurants` is sent, then the response must be 400 with a validation error naming the missing field.
2. Given `?day=2026-05-23&slot=Lunch&wifeApprovedOnly=true`, when the request is sent, then every item in the response must have `wifeApproved=true` and a slot that includes lunch.
3. Given a seeded restaurant 6 minutes from a referenced activity, when `?nearActivityId=<id>` is supplied, then that restaurant must appear earlier in the list than restaurants ≥30 min away.

## L2-020: Local events filter by weekend and drive radius
**Traces to:** L1-007

`GET /api/events` must accept `weekendOf` and `maxDriveMinutes` (default 120) and return events whose start date falls between Friday of that weekend and the following Sunday inclusive.

**Acceptance Criteria:**
1. Given a weekend `weekendOf=2026-05-23`, when `GET /api/events?weekendOf=2026-05-23` is sent, then every returned event must have `startsOn` between 2026-05-22 (Fri) and 2026-05-24 (Sun) or be in a "coming soon" window of ≤14 days after.
2. Given `?maxDriveMinutes=30`, when the request is sent, then no returned event may have `driveMinutes > 30`.

## L2-021: Errand add returns updated weekend
**Traces to:** L1-008

`POST /api/weekends/{weekendId}/errands` must append a `ShoppingErrand` to the weekend and return the updated `WeekendDto`.

**Acceptance Criteria:**
1. Given a weekend with zero errands, when `POST /api/weekends/{id}/errands` is sent with `{description: "Costco run", estimatedMinutes: 45}`, then the response must be 200 with `errands.length=1` and that errand's `done=false`.
2. Given the errand page on `/errand`, when the user submits the form with description + minutes + a "Best day" chip selected, then exactly one POST must be sent and the user must be navigated back to `/weekend` (or `/itinerary`) on success.

## L2-022: Errand done flag flips
**Traces to:** L1-008

`PUT /api/errands/{id}/done` must set the errand's `Done` field to the request value.

**Acceptance Criteria:**
1. Given an errand with `done=false`, when `PUT /api/errands/{id}/done` is sent with `{done: true}`, then a subsequent `GET` of the weekend must show that errand's `done=true`.
2. Given a non-existent errand ID, when the endpoint is called, then the response must be 404.

## L2-023: Weather endpoint returns Sat+Sun forecast for the family's home
**Traces to:** L1-009

`GET /api/weather?weekendOf=<date>` must return exactly two forecast entries (Saturday and Sunday) for the family's home location, with each entry exposing day name, high temperature, low temperature, condition code, and a note string.

**Acceptance Criteria:**
1. Given `weekendOf=<a Saturday>`, when the endpoint is requested, then the response must be a JSON array of length 2 with the first entry's date equal to `weekendOf` and the second entry's date equal to `weekendOf + 1 day`.
2. Given the upstream provider returns 500, when the endpoint is requested, then the response must still be 200 with two neutral/unavailable forecast entries (no client-visible 5xx).
3. Given two requests for the same coordinates and date range within 60 minutes, when the second request fires, then it must be served from cache (verifiable by upstream call count = 1).

## L2-024: Weather provider is configurable and warns on placeholder key
**Traces to:** L1-009, L1-015

The weather provider base URL must come from configuration (`Saturdaze:Weather:BaseUrl`), and on production startup the system must log a warning if any production-required configuration value is unset or still holds its placeholder.

**Acceptance Criteria:**
1. Given the configuration sets `Saturdaze:Weather:BaseUrl="https://example.invalid/"`, when the API starts, then `OpenMeteoWeatherClient`'s `HttpClient.BaseAddress` must equal that URL.
2. Given `ASPNETCORE_ENVIRONMENT=Production` and `SATURDAZE_JWT_SIGNING_KEY` is the placeholder, when the API starts, then a warning log entry must be emitted naming the missing key.

## L2-025: Saved weekend history is paginated and filterable
**Traces to:** L1-010

`GET /api/weekends/history?take=N` must return up to N past weekends ordered by `WeekendOf` descending and include each weekend's title, rating, favourite flag, and a short highlights summary.

**Acceptance Criteria:**

2. Given a request with `?take=0`, when the endpoint is called, then the response must be 400 (zero is not a valid page size).
3. Given the saved page on `/saved`, when the user taps the "Favourites" filter chip, then only `IsFavourite=true` weekends must be displayed.

## L2-026: Weekend favourite and rating toggles persist

`PUT /api/weekends/{id}/favourite` must set `IsFavourite` to the request value, and the system must persist a 1–5 star rating on every weekend.

**Acceptance Criteria:**

## L2-028: All pages render correctly at every breakpoint
**Traces to:** L1-011

Every routed page (`/`, `/weekend`, `/itinerary`, `/activities`, `/restaurants`, `/events`, `/errand`, `/saved`, `/profile`, all auth pages) must render usably at viewport widths 360 px, 576 px, 768 px, 992 px, and 1440 px.

**Acceptance Criteria:**
1. Given viewport 360×800, when any page is rendered, then no element must overflow the horizontal viewport (no horizontal scroll).
2. Given viewport ≥1024 px, when `/weekend` or `/itinerary` is rendered, then the split-view must be in two-column mode with the master pane sticky at the top of the scroll container.
3. Given viewport <1024 px, when `/weekend` or `/itinerary` is rendered, then the master and detail panes must stack into a single column.
4. Given visual baselines captured from `docs/mocks/` with `SD_BASELINE=1`, when the Playwright visual specs run, then each page must match its baseline within the configured pixel-diff tolerance.

## L2-029: Passwords are stored as PBKDF2 hashes
**Traces to:** L1-012

Plaintext passwords must never be written to the database, and the password hash format must use PBKDF2 with at least 100,000 iterations and a per-user random salt.

**Acceptance Criteria:**
1. Given a user registers with password `Passw0rd!`, when the `Users` table is inspected, then the `PasswordHash` column must not contain the substring `Passw0rd!`.
2. Given two users register with the same password, when their hashes are compared, then the hashes must differ (per-user salting).

## L2-030: JWT signing key is loaded from environment, never committed
**Traces to:** L1-012

The JWT signing key must be read from the `SATURDAZE_JWT_SIGNING_KEY` environment variable in production and never embedded as a real value in source control.

**Acceptance Criteria:**
1. Given `SATURDAZE_JWT_SIGNING_KEY` is unset and `ASPNETCORE_ENVIRONMENT=Production`, when the API starts, then a warning log must indicate the signing key is using a placeholder.
2. Given the `appsettings.json` files in source control, when they are inspected, then the `Saturdaze:Jwt:SigningKey` value must be the literal placeholder string and not a real key.

---

## L2-031: CORS origins are explicit allowlist in production
**Traces to:** L1-012

In production, the API must accept cross-origin requests only from the explicit list configured in `Cors:AllowedOrigins`. Wildcard origins must be rejected.

**Acceptance Criteria:**
1. Given `Cors:AllowedOrigins=["https://allowed.example"]` and a CORS preflight from `Origin: https://allowed.example`, when `OPTIONS /api/auth/login` is sent, then the response must include `Access-Control-Allow-Origin: https://allowed.example`.
2. Given a preflight from `Origin: https://evil.example` and that origin is not configured, when `OPTIONS /api/auth/login` is sent, then the response must omit `Access-Control-Allow-Origin` (or return a 403).

---

## L2-032: All database access uses parameterized queries
**Traces to:** L1-012

The system must use EF Core LINQ or parameterized SQL for every database call; no string-interpolated SQL is permitted.

**Acceptance Criteria:**
1. Given the codebase, when grep is run for `FromSqlRaw`, `ExecuteSqlRaw`, or string-interpolated SQL inside C# files under `backend/src/`, then either zero matches must exist or every match must be wrapped with explicit parameter binding (not raw interpolation).

---

## L2-033: Refresh tokens rotate and track creator IP
**Traces to:** L1-012

Every issued refresh token must record `CreatedAtUtc`, `ExpiresAtUtc` (14 days), `CreatedByIp` (when available), and on use must be replaced by a new token with `ReplacedByTokenId` linking back.

**Acceptance Criteria:**
1. Given a user logs in twice in sequence, when the database is inspected, then two `RefreshToken` rows must exist with the second's `Id` referenced by the first's `ReplacedByTokenId` (or both alive if simultaneous-login policy permits multiple sessions).
2. Given a refresh token reaches `ExpiresAtUtc`, when it is presented to the refresh endpoint, then the response must be 401 and the user must be forced to log in again.

---

## L2-034: API responses meet latency budgets
**Traces to:** L1-013

Read endpoints must respond within 500 ms p95 on warm cache; write endpoints within 1000 ms p95; weekend generation within 2000 ms p95.

**Acceptance Criteria:**
1. Given a warmed API, when 50 sequential `GET /api/activities` requests are issued, then the p95 latency must be ≤ 500 ms.
2. Given a warmed API, when 20 sequential `POST /api/weekends/plan` requests are issued for different weekends, then the p95 latency must be ≤ 2000 ms.

---

## L2-035: Angular routes are lazy-loaded
**Traces to:** L1-013

Every routed page (other than the splash) must be lazy-loaded via `loadComponent`, so the initial bundle excludes those pages' code.

**Acceptance Criteria:**
1. Given a production `ng build`, when the build output is inspected, then `dist/saturdaze/browser/` must contain a separate lazy chunk for each of `/login`, `/signup`, `/weekend`, `/itinerary`, `/activities`, `/restaurants`, `/events`, `/errand`, `/saved`, `/profile`.
2. Given the initial bundle, when its size is measured, then it must respect the `500 kB warning / 1 MB error` budget configured in `angular.json`.

## L2-036: Every interactive element is keyboard reachable
**Traces to:** L1-014

Every button, link, form control, and dialog action must be reachable via Tab order and operable with Enter/Space, and the visible focus indicator must meet WCAG 2.1 AA contrast.

**Acceptance Criteria:**
1. Given any routed page, when a user tabs from the top of the document, then every interactive element must receive focus in a logical order and display a visible focus ring.
2. Given an open dialog, when the user presses Tab repeatedly, then focus must be trapped inside the dialog (not escape to the page behind it).
3. Given the user presses Escape while a dialog is open, then the dialog must close and focus must return to the element that triggered it.

---

## L2-037: Every meaningful element exposes an accessible name
**Traces to:** L1-014

Icon-only buttons must carry `aria-label`, form fields must have associated labels, and images carrying meaning must have descriptive `alt` text.

**Acceptance Criteria:**
1. Given any icon-only button (e.g., back arrow, refresh, calendar in the top bar), when inspected, then it must expose a non-empty accessible name via `aria-label` or visually hidden text.
2. Given any text input on the auth pages, when inspected, then it must have a programmatically associated label (matching `for` / `id` or `aria-labelledby`).

---

## L2-038: Color contrast meets WCAG 2.1 AA
**Traces to:** L1-014

All text, including chip labels and disabled controls used to convey state, must meet a 4.5:1 contrast ratio against its background (3:1 for large text ≥18 pt or 14 pt bold).

**Acceptance Criteria:**
1. Given any text node in any page, when axe-core is run, then no contrast violations must be reported.
2. Given any chip showing a state (locked, warn, primary), when its color combination is sampled, then its text/background ratio must be ≥ 4.5:1.

## L2-039: Structured logs are emitted for every request
**Traces to:** L1-015

Every API request must produce a Serilog log entry with method, path, status code, elapsed milliseconds, and (when authenticated) the user ID — but never the bearer token, password, or password hash.

**Acceptance Criteria:**
1. Given any request to `/api/auth/me`, when the log output is inspected, then a single request-completion entry must exist with `method=GET`, `path=/api/auth/me`, a status code, and an elapsed-ms value.
2. Given a login request, when the log output is inspected, then no log line must contain the submitted plaintext password or the issued access/refresh tokens.

---

## L2-040: Upstream provider failures degrade gracefully
**Traces to:** L1-009, L1-015

Upstream provider exceptions (HTTP, JSON, timeout) must be caught, logged at warning level, and substituted with a neutral fallback so the user never sees a 5xx caused by the upstream.

**Acceptance Criteria:**
1. Given Open-Meteo returns 503, when `GET /api/weather` is requested, then the API must respond 200 with two "unavailable" forecast entries and one warning-level log entry referencing the upstream failure.
2. Given a TaskCanceledException from Open-Meteo, when the same request is made, then the same fallback behavior must apply (no unhandled exception, no 5xx).

---

## L2-041: Pushes to main deploy API, run migrations, and deploy frontend
**Traces to:** L1-016

The `.github/workflows/deploy.yml` workflow must fire on every push to `main` and must produce, in order: a successfully deployed API on Azure App Service, applied EF migrations against Azure SQL, and a successfully deployed Angular bundle on Azure Static Web Apps.

**Acceptance Criteria:**
1. Given a commit is pushed to `main`, when the workflow completes, then all three jobs (`api`, `migrate`, `web`) must succeed.
2. Given the `migrate` job runs against an Azure SQL database where all migrations are already applied, when it executes, then the job must succeed without errors (idempotent).
3. Given the `web` job uploads the Angular build output, when the build output is inspected, then it must contain `staticwebapp.config.json` with a navigation fallback to `/index.html`.

## L2-042: SPA navigation fallback serves index.html for deep routes
**Traces to:** L1-011, L1-016

Refreshing any client-side route (e.g., `/login`, `/weekend`, `/itinerary`) on the Static Web App must return the SPA shell, not a 404.

**Acceptance Criteria:**
1. Given the deployed SWA, when `GET https://<swa-host>/login` is requested, then the response must be 200 and the body must contain `<app-root`.
2. Given the deployed SWA, when `GET https://<swa-host>/static/missing-asset.js` is requested, then the response must be 404 (asset-extension paths must not be rewritten).

## L2-043: `saturdaze migrate` applies EF migrations idempotently
**Traces to:** L1-017

The `saturdaze migrate` CLI command must apply pending EF Core migrations to the configured database and must be safe to run repeatedly.

**Acceptance Criteria:**
1. Given an empty database, when `saturdaze migrate --connection <cs>` is invoked, then every migration in `Saturdaze.Infrastructure/Migrations/` must be applied and the `__EFMigrationsHistory` table must list each one.
2. Given a database that already has all migrations applied, when the same command is invoked again, then the process must exit 0 with no schema changes.
3. Given the `--connection` flag is omitted, then the CLI must resolve the connection string from `SATURDAZE_CONNECTION` env var or `ConnectionStrings:Saturdaze` configuration before failing.

## L2-044: `saturdaze seed` is idempotent and scope-aware
**Traces to:** L1-017

The `saturdaze seed` CLI command must seed activities, restaurants, local events, the Brown family, and seed users from bundled JSON, must skip rows that already exist (by natural key), and must accept a `--seed-dir` override.

**Acceptance Criteria:**
1. Given a freshly migrated database, when `saturdaze seed` is invoked, then the response must report inserted counts for each seeded entity and the database must contain those rows.
2. Given a database that already contains the seed rows, when `saturdaze seed` is invoked, then no duplicate rows must be inserted and the process must exit 0.
3. Given `--seed-dir <path>` pointing at a directory with overriding JSON files, when the command is invoked, then values from that directory must take precedence over bundled defaults.

## L2-045: `saturdaze reset` rebuilds a development database
**Traces to:** L1-017

The `saturdaze reset` CLI command must drop the schema, re-apply migrations, and re-seed, gated behind a confirmation step (or `--yes`).

**Acceptance Criteria:**
1. Given any non-empty development database, when `saturdaze reset --yes --connection <cs>` is invoked, then on completion the database must contain only the migrated schema plus the bundled seed data.
2. Given the command is invoked without `--yes` against a non-empty database, then it must prompt for confirmation and abort if not confirmed.
3. Given `ASPNETCORE_ENVIRONMENT=Production` or an equivalent guard, then the command must refuse to run without an explicit override flag to prevent accidental production resets.

## L2-046: Authenticated users can submit a community event
**Traces to:** L1-018

`POST /api/events/submissions` must accept a payload from an authenticated user, persist an `EventSubmission` row with `status="pending"` and `submittedByUserId` set to the caller, and return the created submission. The payload must require `title`, `startsAtLocal` (ISO 8601 local date-time), and optionally accept `location`, `description`, `costNote`, `ageRange`, and `sourceUrl`. Pending submissions must not appear in `GET /api/events` responses for any user.

**Acceptance Criteria:**
1. Given an authenticated user, when `POST /api/events/submissions` is sent with `{title: "Port Credit Buskerfest", startsAtLocal: "2026-06-20T14:00"}`, then the response must be 201, the body must include `status="pending"`, and a `GET /api/events?weekendOf=2026-06-20` from any other user must not include that title.
2. Given an unauthenticated request, when the same endpoint is called, then the response must be 401.
3. Given a payload missing `title` or `startsAtLocal`, when the endpoint is called, then the response must be 400 with a validation error naming the missing field(s).
4. Given a `sourceUrl` that is not a syntactically valid http(s) URL, when the endpoint is called, then the response must be 400 with error code `invalid_url`.
5. Given any successful submission, when the database is inspected, then the row must record `submittedByUserId`, `submittedAtUtc`, and `status="pending"`.

## L2-047: Event submission form defaults date and time to next hour
**Traces to:** L1-018

The "Submit an event" screen and its dialog twin must default the date+time field to one hour from now, rounded up to the next whole hour in the user's local time zone, and must disable the submit button until `title` is non-empty.

**Acceptance Criteria:**
1. Given the current local time is 2026-05-18 11:23, when the user opens the submit form, then the default value of the date-time field must be `2026-05-18T13:00` (now + 1h, snapped up to the next whole hour).
2. Given the current local time is exactly 2026-05-18 11:00:00, when the user opens the submit form, then the default must be `2026-05-18T12:00` (the rule snaps up even when already on the hour, to preserve the +1h offset).
3. Given the form is freshly opened with only the default date-time, when no title is entered, then the "Submit event" button must be disabled and announce `aria-disabled="true"`.
4. Given a non-empty `title` is entered, when all other fields are left blank, then the submit button must become enabled and submitting must succeed (per [L2-046](#l2-046-authenticated-users-can-submit-a-community-event)).
5. Given the user clears the date-time field, when they attempt to submit, then an inline validation error must appear on the field and submission must be blocked.

## L2-048: Submission entry points exist on the events feed
**Traces to:** L1-018

`/events` must surface two affordances to submit a new event: a primary floating "+" button anchored bottom-right that opens the quick-add dialog, and a secondary text/icon action in the page header that navigates to the full `/events/submit` screen. The FAB must remain visible while scrolling and must not overlap the bottom navigation or its tablet/desktop left rail.

**Acceptance Criteria:**
1. Given the user is on `/events`, when the page is rendered at ≤575 px wide, then a circular "+" button with `aria-label="Submit a new event"` must be positioned above the bottom navigation and remain fixed while the feed scrolls.
2. Given the user is on `/events` at ≥720 px wide, when the page is rendered, then the FAB must sit clear of the left-rail navigation (≥16 px gap) and remain anchored to the bottom-right of the content column.
3. Given the user taps the FAB, when the action sheet opens, then the quick-add dialog must open in place over the feed; given the user taps "Open full form" inside it, then the app must navigate to `/events/submit`.
4. Given the user taps the secondary header action "Submit an event", when activated, then the app must navigate directly to `/events/submit` without first opening the dialog.

## L2-049: Pending submissions are visible to their submitter only
**Traces to:** L1-018

`GET /api/events/submissions/mine` must return the caller's own submissions in any status (`pending`, `approved`, `rejected`). The events feed at `/events` must render a "Pending review" badge on cards drawn from the caller's own pending submissions so they understand the event isn't yet public.

**Acceptance Criteria:**
1. Given the user has submitted an event that is still pending, when `GET /api/events/submissions/mine` is called, then the response must include that submission with `status="pending"`.
2. Given another authenticated user, when they call `GET /api/events/submissions/mine`, then the first user's pending submission must not appear in the response.
3. Given the events page on `/events`, when the submitter loads the page, then their own pending submission must render as an `sd-event-card` with a `Pending review` chip and must not be tappable to "Add to weekend".
4. Given a non-submitting user loads `/events`, when the page renders, then no pending submission belonging to anyone else must be present in the DOM.

## L2-050: Admin moderation queue approves or rejects submissions
**Traces to:** L1-018

Users with the `admin` role must have access to `/admin/events` listing all pending submissions. Approving a submission must move its `status` to `approved` and copy it into the public events catalogue; rejecting must move its `status` to `rejected` and record an optional `rejectionReason`. Non-admins must receive 403 when calling the moderation endpoints.

**Acceptance Criteria:**
1. Given an admin user with one pending submission, when they `POST /api/events/submissions/{id}/approve`, then the submission's `status` must become `approved` and the event must subsequently appear in `GET /api/events` for any user whose weekend covers its date.
2. Given an admin user, when they `POST /api/events/submissions/{id}/reject` with `{reason: "duplicate of seeded event"}`, then `status` must become `rejected` and `rejectionReason` must be persisted.
3. Given a non-admin authenticated user, when they call either moderation endpoint, then the response must be 403 with error code `forbidden`.
4. Given an admin loads `/admin/events`, when the page renders, then it must list every `pending` submission with submitter email, submitted timestamp, and the event's proposed title/date/time, sorted oldest-first.
5. Given the admin taps "Reject" on a row, when the confirmation dialog opens and they enter a reason and confirm, then exactly one `POST /api/events/submissions/{id}/reject` call must be made and the row must disappear from the queue on success.
