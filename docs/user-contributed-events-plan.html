User-Contributed Local Events — Implementation Plan

Traces to L1-018 and L2-046 → L2-050 in docs/specs/. Mocks: docs/mocks/pages/events.html (FAB added), events.submit.html (new), events.submitted.html (new), admin.events.html (new), dialogs.html (M-D13/14/15).

Context

Saturdaze currently has a curated LocalEvent catalogue seeded from local-events.json and served read-only at GET /api/events. There is no way for a family to tell Saturdaze about something the system doesn’t already know — a school fair, a Buskerfest, a one-off concert — even though those are exactly the kind of events the Browns would want in their weekend.

L1-018 closes that gap: any authenticated user can submit an event; admin approval gates publication so the curated feed stays clean; the submitter sees status in their own feed without any new notification plumbing.

Why now: the events page is otherwise feature-complete, but its value is bounded by what the seed list happens to cover. User contribution is the cheapest way to grow coverage without paid editorial work.

Confirmed design choices (from clarification):

  1. Notify approach — status visible in the submitter’s own feed only (no email / no notifications table).
  2. Admin entry point — admin-only link inside the Profile page when role === 'Admin'; bottom-nav stays four items.
  3. Data model — separate EventSubmissions table; on approve, copy fields into the existing LocalEvents table (public read path unchanged).

Step 2 — Application layer (CQRS handlers + DTOs)

Files to add under backend/src/Saturdaze.Application/EventSubmissions/:

  • EventSubmissionDto.cs — flat record returned from all endpoints.
  • SubmitEventCommand.cs + SubmitEventCommandHandler.cs — covers L2-046.
    • Validates Title non-empty, StartsAtLocal parses, SourceUrl is http(s).
    • Sets Status = Pending, SubmittedByUserId from ICurrentUserAccessor.
  • ListMySubmissionsQuery.cs + handler — covers L2-049 (GET /api/events/submissions/mine).
  • ListPendingSubmissionsQuery.cs + handler — admin queue, oldest-first.
  • ApproveSubmissionCommand.cs + handler — sets status, creates a LocalEvent row, sets PublishedEventId. Idempotent: re-approving an already-approved row is a no-op.
  • RejectSubmissionCommand.cs + handler — sets status + reason; cannot reject an already-approved row (returns 409).

Reuse:

  • ICurrentUserAccessor from backend/src/Saturdaze.Api/Authentication/HttpContextCurrentUserAccessor.cs (already injected via Saturdaze.Application.Abstractions.ICurrentUser).
  • The existing LocalEvent entity for the publish-on-approve copy.

Skip: no MediatR pipeline behaviours beyond what’s already wired; no validator library — the handlers throw the existing ValidationException like other commands do.

Step 3 — API controller

File to add: backend/src/Saturdaze.Api/Controllers/EventSubmissionsController.cs.

Routes:

Verb Route Authorize Handler
POST /api/events/submissions [Authorize] SubmitEventCommand
GET /api/events/submissions/mine [Authorize] ListMySubmissionsQuery
GET /api/events/submissions/pending [Authorize(Policy = "Admin")] ListPendingSubmissionsQuery
POST /api/events/submissions/{id}/approve [Authorize(Policy = "Admin")] ApproveSubmissionCommand
POST /api/events/submissions/{id}/reject [Authorize(Policy = "Admin")] RejectSubmissionCommand

The "Admin" policy already exists in Program.cs (lines 75–78) but is currently unused — this is its first consumer.

No change to the existing EventsController.GetEvents — the public feed continues to read from LocalEvents, so approved-and-copied submissions surface automatically.

Verification — end-to-end

Run these after Step 9 in order:

  1. Backend unit + integration tests — cd backend && dotnet test. Expect green; new tests traced to L2-046/049/050.
  2. Migration applies cleanly — saturdaze migrate --connection <localdb>; inspect EventSubmissions table exists with the indexes from Step 1.
  3. Seed — saturdaze seed; verify the admin user and 3 pending rows.
  4. Frontend lint + build — cd frontend && npm run lint && npm run build.
  5. Frontend unit tests — npm test --workspace=saturdaze — nextHourFromNow cases pass for L2-047 #1 and #2.
  6. Run the app — start API + Angular; sign in as the admin seed user.
    • Navigate /events. Tap the FAB → quick-add dialog opens (M-D13). Submit with only a title → lands on /events/submitted. Back to /events → see “Pending review” chip on own card.
    • Sign in as the non-admin Brown user — confirm that submission is not visible.
    • As admin → Profile → “Event moderation”. Approve the row → confirm dialog (M-D14) → row disappears.
    • Sign back in as the Brown user → that event now appears in the regular Saturday section.
  7. Playwright behaviour suite — cd e2e && npm test. Specs traced to L2-046 through L2-050 all pass.
  8. Visual baselines — cd e2e && npm run baseline; commit only the snapshot deltas for the new screens + the events page update.
  9. Manual a11y pass — keyboard-only flow through submit dialog and admin reject dialog: focus is trapped, ESC closes, submit button announces aria-disabled="true" when title is blank (per L2-047 #3 + L1-014).

If all nine pass, the feature is shippable.

Out of scope (deliberate)

  • Notification table, email delivery, push.
  • A separate admin shell or fifth nav item.
  • Drive-time auto-calculation on submission (the admin can set DriveMinutes when approving — or leave it null for the public feed to omit the drive chip).
  • Editing or deleting one’s own submission. If the submitter realises a typo, they re-submit and the admin rejects the duplicate.
  • Image / photo upload for the event. Defer.
